# Migrating from 0.3.x to 0.4.0

**Status:** active migration guide for the 0.4.0 release.
**Roadmap:** `docs/design/path-a-roadmap.md` §3.3, §9.9, §9.10, §13.

---

## TL;DR — pick your install line

| You are… | Install command |
|---|---|
| An **embedding host** (FastAPI, Slack bot, pytest fixture, …) — you call `Agentao(...)` from Python and want the smallest closure | `pip install agentao` |
| A **CLI user** — you run the `agentao` command in a terminal | `pip install 'agentao[cli]'` |
| **Already on 0.3.x** and you want zero behaviour change | `pip install 'agentao[full]'` |

If you upgrade from 0.3.x with plain `pip install -U agentao` and you also use the CLI, the `agentao` command will now exit 2 with the message:

```
agentao CLI requires extra packages (missing: rich).
  pip install 'agentao[cli]'   # CLI surface only
  pip install 'agentao[full]'  # 0.3.x-equivalent closure
See docs/migration/0.3.x-to-0.4.0.md for details.
```

That message is the safety net (P0.10). Run one of the install commands it prints and you are done.

---

## Why the change

In 0.3.x, `pip install agentao` pulled 13 direct dependencies including `rich`, `prompt-toolkit`, `readchar`, `pygments`, `beautifulsoup4`, and `jieba`. For a Python project that only embeds Agentao via `from agentao import Agentao`, those CLI / web-fetch / Chinese-tokenization libraries were dead weight in `pip freeze` — and harder to justify when the host already had its own UI / web / i18n stack.

0.4.0 splits them into opt-in extras so the embedded persona pays only for what they use. The CLI persona is one extra away.

The split is the **only** breaking change in the entire Path A P0 plan (`docs/design/path-a-roadmap.md` §3). All other 0.3.x → 0.4.0 migrations are pure additive — no API rename, no constructor signature change, no behaviour change inside `Agentao.chat()`.

## Concrete dependency map

| Direct dep in 0.3.x | Where it lives in 0.4.0 |
|---|---|
| `openai` | core |
| `httpx` | core |
| `pydantic` | core |
| `pyyaml` | core |
| `mcp` | core |
| `python-dotenv` | core |
| `filelock` | core (lazy-imported per P0.5; required for skill registry mutations) |
| `rich` | `[cli]` |
| `prompt-toolkit` | `[cli]` |
| `readchar` | `[cli]` |
| `pygments` | `[cli]` |
| `beautifulsoup4` | `[web]` |
| `jieba` | `[i18n]` |

Existing extras are unchanged: `[pdf]`, `[excel]`, `[image]`, `[crypto]`, `[google]`, `[crawl4ai]`, `[tokenizer]`. `[full]` now meta-extras over `cli + web + i18n + all of the above`, so `pip install 'agentao[full]'` reproduces the 0.3.x bundled closure exactly. The exact-closure equivalence is enforced in CI by `tests/test_dependency_split.py` against `tests/data/full_extras_baseline.txt` (122 packages, frozen 2026-05-01).

## How to update common project shapes

### A. `pyproject.toml` of an embedding host

Before (0.3.x):

```toml
[project]
dependencies = [
  "agentao>=0.3.0",
  # …
]
```

After (0.4.0):

```toml
[project]
dependencies = [
  "agentao>=0.4.0",     # core only — minimal closure
  # …
]
```

If you actually use the web-fetch tool or Chinese-tokenization memory:

```toml
dependencies = [
  "agentao[web,i18n]>=0.4.0",
  # …
]
```

If you are not sure what you use, start with `agentao>=0.4.0` and let an `ImportError` at runtime point at the missing extra. Each runtime path that needs an extra has been marked lazy in P0.5, so the failure mode is a clear `ImportError: bs4` (or similar) the first time you call the tool — not a silent wrong-answer.

### B. CLI users (`pip install agentao` from a shell)

Before:

```bash
pip install agentao
agentao --help
```

After:

```bash
pip install 'agentao[cli]'
agentao --help
```

Note the quotes — bracket extras need to be quoted in most shells.

### C. CI pipelines

If your CI installs Agentao and runs the `agentao` CLI (e.g. for a smoke check), pin `[cli]`:

```yaml
- run: pip install 'agentao[cli]==0.4.0'
- run: agentao --print "hello"
```

If your CI only constructs `Agentao()` from Python and never invokes the console script, the bare install is enough:

```yaml
- run: pip install 'agentao==0.4.0'
- run: python -c "from agentao import Agentao; ..."
```

### D. "I want zero change"

```bash
pip install 'agentao[full]>=0.4.0'
```

Your `pip freeze` will match the 0.3.x closure to the package level. The only difference you may observe is patch-level version drift in transitive deps, which `pip` would have introduced anyway over normal upgrade cadence.

## Known follow-ups (post-0.4.0)

- `python-dotenv` is currently in core because the `.env` discovery path is touched on construction in some embedded shapes. If P0.5's lazy-import work covers this fully, `python-dotenv` will move to a `[dotenv]` extra in 0.4.1.
- `filelock` is lazy-imported per P0.5 but still in core because the skills registry mutation path needs it. If a future patch makes the skills registry purely read-only by default for embedded hosts, `filelock` may move to a `[skills-write]` extra.

Both follow-ups are **patch-level, additive** (they would only ever shrink the core, never expand it). Neither requires a 0.5.0 break.

## If something breaks

1. Reproduce in a fresh venv against the published 0.4.0 wheel.
2. Capture the failing command + traceback (or the friendly message + which install line you ran).
3. File an issue at https://github.com/jin-bo/agentao/issues with label `0.4.0-migration`.
4. As an immediate workaround: `pip install 'agentao[full]==0.4.0'` reproduces the 0.3.x closure exactly. That is the documented "no-change" install line and unblocks any path the migration may have missed.

If within 48 hours of 0.4.0 hitting PyPI two or more such issues arrive without `[full]` resolving them, the release will be yanked per the §13.3 rollback protocol. That preserves the "zero public-API breaks" metric (§2.1) — a packaging mismatch is a packaging mismatch, not an API break, and a yank-and-redo round trip is cheaper than eroding the metric.
